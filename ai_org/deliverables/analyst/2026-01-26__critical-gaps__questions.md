## Раздел 2: Недостающие сущности

### Q2.1 Заявка на улучшение — детальные требования ✅ ОТВЕТ ПОЛУЧЕН

**ОТВЕТ ВЛАДЕЛЬЦА ПРОДУКТА:**

**Назначение:**
Механизм подачи и отбора инициатив сотрудниками с последующей трансформацией
одобренных инициатив в проекты с бюджетом и ответственными.

**Статусы (фиксированная цепочка):**
1. Draft — черновик (инициатор) (описание проекта, презентация)
2. Submitted — подана
3. Screening — первичный скрининг (владелец процесса / бизнес-куратор)
4. Evaluation — оценка (аналитик/продукт + ИТ/архитектор + финансы)
5. Manager Approval — согласование руководителя инициатора
6. Committee Review — рассмотрение на портфельном/инвесткомитете (на этом этапе должен быть паспорт проекта)
7. Budgeting — резервирование/утверждение бюджета
8. Project Created — инициирована как проект
9. In Progress / Completed — дальнейшее управление в контуре проекта

**Роли:**
- Initiator — инициатор (любой сотрудник)
- Process Owner / Business Curator
- Analyst / Product Owner
- IT / Architecture
- Finance / Controlling
- Line Manager (инициатора)
- Committee
- PMO / Project Manager (после одобрения)

**Связь с проектом:**
- После статуса "Committee Approved" создаётся Project (связь 1:1)
- В проект автоматически переносятся:
  - цель
  - ожидаемый эффект (KPI / финансовый эффект)
  - спонсор
  - владелец
  - сроки
  - утверждённый бюджетный лимит
  - все документы включая паспорт
  - гантт
- В проекте автоматически создаётся первичный бэклог (эпики/задачи)

### Q2.3 Планирование ресурсов — модель расчёта ✅ ОТВЕТ ПОЛУЧЕН

**ОТВЕТ ВЛАДЕЛЬЦА ПРОДУКТА:**

**Норма часов:**
- Источник: производственный календарь + корпоративная норма (8 часов/день)
- Планирование ведётся в FTE и/или часах в неделю по ролям

**Отсутствия:**
- Учитываются обязательно:
  - отпуска
  - праздники
  - out-of-office
- Минимальная реализация: параметр "доступность (%)" в профиле сотрудника
- Целевая реализация: автоматическое получение отсутствий из Exchange / EWS

## Раздел 4: UI/UX решения

### Q4.1 Навигация — окончательная структура ✅ ОТВЕТ ПОЛУЧЕН

**ОТВЕТ ВЛАДЕЛЬЦА ПРОДУКТА:**

**Источник истины:**
- Источником истины по контекстной навигации и структуре экранов являются следующие макеты:
  - 01-tasks.html — Задачи
  - 02-service-desk-portal.html — Service Desk (User)
  - 03-service-desk-agent.html — Service Desk (Agent)
  - 04-meetings.html — Встречи (обзор)
  - 05-meeting-script-recording.html — Встреча (проведение)
  - 06-meeting-transcription.html — Встреча (результаты)

**Global Navigation (верхняя панель) — БИЗНЕС-РЕШЕНИЕ:**
Верхняя панель НЕ полностью отражена в макетах и задаётся отдельным бизнес-решением.

**Финальный состав Global Navigation (Top Bar):**
1. Главная (ex-Dashboard)
2. Сотрудники
3. Задачи
4. Встречи
5. Почта
6. Сообщения
7. SD (Service Desk)
8. Аналитика

**Правила (обязательные):**

1. Верхняя панель (Global Navigation):
   - едина для всех экранов
   - всегда содержит полный список глобальных разделов
   - не зависит от текущего контекста (проект, встреча, заявка и т.д.)

2. Левый сайдбар (Context Navigation):
   - формируется строго по макетам соответствующего раздела
   - содержит ТОЛЬКО контекстные пункты
   - меняется при переходе между разделами верхней панели

3. **Запрещено:**
   - дублировать пункты Global Navigation в левом сайдбаре
   - размещать Почту / Сообщения / SD / Аналитику в контекстном сайдбаре задач или встреч

**Интерпретация:**
- Если пункт есть в Global Navigation, но отсутствует в конкретном макете —
  он всё равно должен присутствовать в верхней панели.
- Если пункт есть в макете сайдбара, но относится к глобальному уровню —
  это считается дефектом реализации.
